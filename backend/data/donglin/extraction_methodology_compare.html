0. 一句话总结

方法 A · 我们的 三段流水线：机器读 → 人翻译 → 规则化编码

把 442-sheet 模板的语义（科目→区域、规则→准则、底稿→认定）手工编码进 Python 字面量。强项：业务语义丰富，能跑出 4,250 单元格追溯。弱点：可重复性差，公式没拉。

方法 B · v3 的 证据驱动：把 Excel 公式当本体一等公民

把 442-sheet 内每一个公式单元格抽成 JSONL（106,614 行），每条本体三元组都强制 :hasEvidenceRef "cell_formulas.jsonl#L14614"。强项：杜绝臆想，公式级精确；弱点：业务语义只到"A 取自 B"，准则映射缺。

3. 核心差异 — 怎么把"审计语义"拿出来

A 语义来源 = 442-sheet 索引体系 + CSA 准则映射 + 审计师 expertise

认为底稿模板的索引体系本身就是审计语义的载体：

  • A6 → 应收账款审定表 这种"A=资产 / 6=应收账款"的对应是人工 hard-code 的 code_to_area 字典
  • 规则 AR-RULE-001 既有自然语言 IF customer.closing_cr>0 THEN 重分类至 2203，又有形式化 closing_cr > 0 表达式 — 两套
  • 规则来源标 "事务所" / "甲所" / "2024-01" — 显式区分公共准则 vs 内部沉淀
  • 每个调整分录 Z6-AI-A6-01 都带 triggered_by_rule 字段反查

语义注入点：extracted_knowledge/0X_*.md 的人工编辑环节。审计师在这里决定"哪一行是 Y3 重要性、Y8.3 留空意味着什么"。

B 语义来源 = Excel 公式 + VBA 赋值 + cross-sheet 引用图

把底稿公式本身视为审计师早已沉淀的语义：

  • A6!C11 = 'A6-2'!F824 这条 Excel 公式直接就是一条 :FillRule 本体三元组（cross-sheet kind）
  • VBA 里的 Sheets("A6").Range("C7").Value = Sheets("KM").Range("B12").Value 是另一条 :FillRule（vba-copy kind）
  • SUMIFS 公式 + 中文字面量 "应收账款" → 推出 :Account hasAccountCode "1122"
  • 每条本体三元组都强制 :hasEvidenceRef "cell_formulas.jsonl#L14614"^^xsd:anyURI — 不能臆想

语义注入点：不在抽取里，在设计 schema 时人工选择哪些公式抽象成哪个 Class。Class 决策也带 EVIDENCE 行号。

关键洞察： 两种方法的"审计语义"侧重不同 —— A 强调业务含义（哪个数对应哪个准则的哪条）， B 强调结构传播（这个数字怎么从原始 KM 表流到最终 ZK 披露）。 A 适合做 Agent 填表 + 规则触发；B 适合做底稿勾稽 + 报表自动汇总验证。

7. 互补关系

A 的优势

  • 📜 14 条业务规则有形式化表达式 — Agent 可程序化求值并触发调整分录
  • 🧠 方法论显式 — CSA 1221 四级梯度 / FIFO 账龄 / 关联担保识别都有专门 OT
  • 🎯 认定 (Assertion) 一等公民 — 6 种认定都是 ObjectInstance，与底稿/规则双向链接
  • 💼 项目维度完整 — Engagement + ProjectTeam + ReviewLevel + ProjectTimeline + AuditedEntity（涵盖业务约定到签发）
  • 🤖 能跑 Agent 填表 — agent_fill.py 1500 行已实证 5 张底稿端到端

A 的弱点

  • ❌ 不可重跑 — 关联担保 5 笔等是字面量编码，新公司要重写
  • ❌ 没用 Excel 公式 — A6!C11='A6-2'!F824 这种沉淀完全没读
  • ❌ 没读 VBA — 甲所模板里 369 条 VBA 映射没用上
  • ❌ 没用 W3C 标准 — JSON 而非 Turtle/OWL，不能 SPARQL 查询
  • ❌ 没有 validate — 字面量错了到运行才发现

B 的优势

  • 🔍 可机器校验的证据链 — 每条三元组指回 jsonl 行号
  • 🔁 可重跑 — 新公司换一套 .xlsm 重跑 extract_excel.py 即可
  • 📐 W3C 标准 — Turtle + SPARQL + SHACL 生态全开
  • 🏗 明确的多团队工序 — Probe / Critique / Extract / Design / Implement 5 阶段
  • ✅ validate.py 18 个 check — bilingual labels / domain-range / individual-class 三联
  • 📊 106K 公式 + 29K 跨表边 — 报表勾稽自动验证基础

B 的弱点

  • ❌ 业务语义薄 — 知道"A 来自 B"，不知道"这是 CSA 1311 应收账款认定"
  • ❌ 不能驱动 Agent 填表 — FillRule 只描述静态映射，无 IF/THEN 业务逻辑
  • ❌ 没有调整分录推理 — Z6-AI-A6-01 / Z6-AI-A9-01 这种规则触发的调整
    缺位
  • ❌ 不读凭证 + 辅助账 — 客户层数据全空
  • ❌ 报告 PDF 完全没用 — 关联担保 / 实控人持股 / 项目组等业务事实在底稿外
🎯 互补关系一句话： A 是"业务驱动的本体"（能跑 Agent，能触发规则，能生成调整分录）， B 是"证据驱动的本体"（每条都可机器追溯，可重跑，可校验）。 理想的 v4 = 用 B 的抽取管线 + 证据链 + Turtle 序列化， 叠 A 的认定/准则/规则 IF-THEN 业务层。

8. 我们可以从 v3 借鉴什么

  1. 跑一遍 extract_excel.py 把 WP_FSR.xlsm 442 sheets 的 106K 公式抽出来，给每个填好的单元格补 :hasEvidenceRef。
  2. VBA 映射进 ontology — 369 条 VBA 赋值规则放进 fill_rules 集合，让 RulesPanel 显示更多。
  3. 引入 Turtle/OWL 序列化层 — JSON 是落库格式，Turtle 是审计师 review 格式。两套并存：ontology/abox.json (用) + ontology/abox.ttl (review)。
  4. validate.py 写 18 个 check — bilingual label / domain-range / individual must-have-class，运行时强制。
  5. Schema 决策表带 EVIDENCE 行号 — 每个新加的 OT 都写 EVIDENCE: cell_formulas.jsonl#L 防臆想。
  6. 多团队工序 — Probe / Critique / Extract / Design / Implement 五段，每段独立产物可 review。

9. v3 可以从我们借鉴什么

  1. AuditRule.formal_expression — 在 FillRule 之外加一层"业务规则"，每条带可程序化求值的 closing_cr > 0 表达式。
  2. Assertion 一等公民 — 6 种认定（Existence / Completeness / Accuracy / Cutoff / RightsAndObligations / Presentation）应该是 NamedIndividual + 与 Rule/Procedure 双向链接。
  3. 调整分录推理 — Z6 调整分录应能从 Rule 触发自动生成（AR-RULE-001 ⇒ Z6-AI-A6-01 的因果链）。
  4. 项目维度 — ProjectTeam + ReviewLevel + ProjectTimeline 补全审计治理结构。
  5. 方法论 OT — FIFO 账龄、CSA 1221 四级梯度、关联担保识别等审计方法应建模为 ObjectType。
  6. 客户数据集成 — 把 TB.xlsx / Aux.xlsx / Vouchers.xlsx 接进来，让"A6.C11 取自 A6-2.F824"进一步追溯到客户原始数据。
