审计本体的 4 层架构 · 跨所复用与差异化策略

基于你的分层思路：把本体知识库按「国际准则 → 中国本地化 → 事务所内规 → 审计师经验」四层组织，TBox 来自上面两层（跨所通用），ABox / Rule / Action 参数来自下面两层（差异化资产）。本文档把现有 2 套抽取方法映射到 4 层模型，并给出 v4 演进路径。

§1 4 层模型 — 标准的审计知识分层

你提出的 4 层模型本质是经典的 Reference Architecture for Audit Knowledge，国际上 Big 4（PwC Aura / Deloitte EMS / EY Helix / KPMG Clara）都是这套思路：

L1 国际通用准则 · ISA / IFRS / IFAC ~30%
全球任何 Big 4 + 国内 8 大都遵守。例：6 种认定 / 审计动作分类 / 工作底稿概念
L2 中国本地化准则 · CSA / CAS / 中注协 / 财政部 ~20%
所有中国所通用。例：CSA 1221 重要性梯度 / CAS 36 关联方 / 中国 4 类审计意见
L3 事务所内规 · Firm-Specific Methodology ~35%
甲所 vs 立信 vs 信永中和 不同。例：甲所的三级复核 / Z6 调整分录编号 / AR-RULE-001 应收贷方重分类
L4 审计师个人经验 · Tacit Knowledge ~15% (现在=0)
最难捕获也最值钱。资深审计师的"直觉判断"、"特殊客户怎么处理"。现在数据为零，未来从 ChatPanel 对话日志累积。
核心论断 — 你提的对：
  • TBox（对象类型骨架）来自 L1+L2 — 这层每个所大概率一样，做一次跨所复用
  • ABox（实例值）+ Rule formal_expression + Action 参数来自 L3 — 这是事务所差异化资产
  • L3 偶尔触发新 OT（如甲所强调"关联担保" → 单独建 RelatedPartyGuarantee）
  • L4 通过 Agent 交互逐步累积 — 复核批注 / 对话日志 / 手动修改 Agent 填的内容

2.1 ObjectType 归位

1L1 国际通用 (~18 个)

ISA / IFRS 框架，跨所跨国通用

  • AuditedEntity / ClientGroup / RelatedParty / Contact
  • Engagement / AuditType / AuditPeriod / Engager
  • ProjectTeam / Auditor / ProjectTimeline
  • TrialBalance / Account / SubLedger / Voucher
  • WorkingPaper / PaperTemplate / AuditReport
  • Assertion (Existence / Completeness / Accuracy / Cutoff / RightsObligations / Presentation)
  • AuditProcedure / AuditEvidence / Anomaly
  • 11 个 ActionType (Read / Map / Standardize / Sample / Reconcile / Recompute / Explain / Generate / Review / SignOff / Archive)
2L2 中国本地化 (~12 个)

CSA / CAS / 国家法律强制

  • MaterialityLevel 含 PM/TE/SUM 三级 → CSA 1221 中国版
  • RiskOfMaterialMisstatement → CSA 1141 中国对 RMM 的特殊要求
  • RelatedPartyRisk → CAS 36 关联方判定（比 IAS 24 更严）
  • AccountingPolicy → 中国"企业会计制度" + "企业会计准则" 双轨
  • AuditOpinion → 中国 4 种意见类型
  • ReviewLevel → 中注协要求三级复核体系
  • AuditedEntity.is_high_tech / is_small_micro / 老准则/新准则 等中国特有字段
3L3 甲会计师事务所专有 (~22 个)

从 WP_FSR.xlsm 442-sheet 模板抽出

  • AuditArea (按 A/B/C/D/G/H 字母分组 9 大区域) — 甲所习惯
  • PaperTemplate.sheets 6 个固定子表 — 甲所模板结构
  • RelatedPartyGuarantee — 甲所单独建类 (其他所可能合并到 RelatedParty)
  • ProjectTimeline 5 个里程碑 — 甲所项目治理
  • Z6 调整分录编号体系 (Z6-AI-* 前缀) — 甲所底稿规范
  • 规则层 7 条事务所内规：AR-RULE-001 / AP-RULE-001 / RP-RULE-001 / INV-RULE-001 / TAX-RECLASS-001 / GC-INDICATOR-001 / DL-RULE-001
4L4 审计师个人经验 (0 个)

现在没有数据，但已有捕获管道

  • 资深审计师的"直觉判断"
  • 特殊客户的处理偏好
  • 对低风险事项的简化做法
  • 跨所跳槽时能"带走"的经验资产

🔮 未来管道：ChatPanel 对话日志 → 提取"审计师手动修改 Agent 填的内容" → 累积成 ProposedRule → ≥ 3 次 → 推荐升级为 L3 规则

§3 两种抽取方法在 4 层模型中的位置

方法 A · 我们 「Top-down 自上而下」

先按 L1+L2 准则手工设计 TBox 骨架，再用 L3 甲所内规填 ABox + Rule

L1 35%
L2 25%
L3 40%
国际 中国 甲所
  • L1+L2 TBox 设计：手工 + LLM 候选 + Palantir 范式 → 52 OT
  • L3 ABox + Rule：人工读 10 个 markdown 知识卡，编码到 build_instances.py
  • 已显式标 source 字段：14 条规则区分"中注协" vs "甲所" → 已在做 L2/L3 区分
  • L4 缺位：无捕获管道

方法 B · v3 「Bottom-up 自下而上」

从甲所 .xlsm 公式 + VBA 自动抽证据，反推应该有哪些 TBox + FillRule

L1 20%
L2 10%
L3 70%
国际 中国 甲所
  • L1+L2 TBox：弱 — 20 Class 是从 442-sheet 反推的，不是从准则开始
  • L3 ABox + FillRule：超强 — 106,614 公式 + 369 VBA 全自动抽，每条带 jsonl 行号
  • 没有 source 区分：FillRule 只有 hasRuleKind (cross-sheet / vba-copy / manual)，不区分 L1/L2/L3
  • L4 缺位：同 A

3.1 映射到分层策略的具体角色

你的分层策略 A 怎么处理 B 怎么处理
L1+L2 定义 TBox ✅ 主力工具 — 手工设计 + LLM 候选生成 + 专家校对 ⚠️ 通过反推得到 — 看 .xlsm 公式存在性 → 反推该有什么 Class
L3 填 ABox + Rule ✅ 显式标 source="甲所" 7 条内规 ✅ 每条都有 evidenceRef: jsonl#L<n>
L3 触发新 OT ⚠️ 靠人工 review，没机器告警机制 ✅ 公式扫不出来的 sheet 模式 → 不建对应 Class（机器约束）
L1+L2/L3 区分标签 ⚠️ 规则有 source 字段，但 OT 没有 _layer 字段 ❌ 全是平的 — 不区分通用 vs 事务所
L4 经验捕获 ❌ 无管道（但 ChatPanel 是潜在入口） ❌ 无管道
跨所迁移 ⚠️ 需手工挑出 L3 重新编码 ✅ 跑同一套 extract_excel.py 在新所 .xlsm 上即可
洞察：两种方法在 4 层模型里是"互补的"，不是"竞争的"
  • 方法 A 强在 L1+L2 设计 — 准则映射、形式化规则、Agent 业务逻辑
  • 方法 B 强在 L3 抽取 — 真实模板沉淀、可重跑、证据链可追
  • 理想 v4 = A 的 TBox 设计 + B 的 L3 抽取管线 + 新增 L4 经验捕获

§4 v4 演进路径 — 把分层策略落到产品

4.1 给每个 OT / Rule 加 _layer 字段

// object_types.json — 加 _layer 字段，表明跨所通用还是事务所专有
{
  "code": "AuditedEntity",
  "_layer": "L1",                  // 国际通用
  "_cluster": "客户类"
},
{
  "code": "MaterialityLevel",
  "_layer": "L2",                  // 中国本地化 — CSA 1221 PM/TE/SUM
  "_cluster": "风险"
},
{
  "code": "AuditArea",
  "_layer": "L3:donglin",         // 甲所按字母分组 9 大区域
  "_cluster": "审计区域"
}

// 规则同样
{
  "code": "AR-RULE-001",
  "_layer": "L3:donglin",
  "_layer_basis": "CAS 22 金融工具列报 (L2 底层) + 甲所 2024-01 形式化阈值 (L3)",
  "source": "甲所"
}

4.2 前端按层级过滤

本体管理页加 「来源层级」过滤胶囊条： L1 国际 L2 中国 L3 甲所 L4 经验 审计师一眼看清"哪些是通用知识/哪些是我们专有"。

4.3 跨所迁移流程

1复用 L1+L2
一字不改 — 准则不变
2替换 L3
甲所 7 内规 → 新所 N 内规
3跑方法 B 抽取
新所 .xlsm 扫公式 + VBA
4证据对账
L3 OT 在新所 .xlsm 找不到证据 → 标"未沉淀"
5机器推荐
高频模式但 L3 没覆盖 → 建议新建

4.4 L4 经验捕获管道 (创新点)

ChatPanel 对话 → 审计师手动改 Agent 填的格
       ↓
   提取 "用户改写事件" + 改写理由 (从对话上下文 LLM 抽)
       ↓
   累积成 ProposedRule 候选 (写到 LearningInbox)
       ↓
   累积 ≥ 3 次类似改写
       ↓
   推荐升级为 L3 规则 (审计师 sign-off)
       ↓
   写入 AuditRule (source="审计师 X 经验", _layer="L3 → L4 沉淀")

这个管道把 "审计师脑子里的隐性知识" 变成 "事务所可继承的本体资产"。审计师跳槽时是否能带走这部分 — 也成为事务所人才战略的议题。

§5 给项目经理的 review 话术

📢 跟项目经理开场白

"侯老师，我抽出来的本体不是平面清单，是 4 层结构：

① L1 国际通用 (~30%) — ISA / IFRS 框架，跨所跨国都适用。包括 6 种认定、审计动作分类、底稿概念。
② L2 中国本地化 (~20%) — CSA / CAS / 中注协 / 财政部。包括 PM/TE/SUM 重要性梯度、关联方判定、4 类审计意见。
③ L3 甲会计师事务所内规 (~35%) — 这是从你给我的 WP_FSR.xlsm + 5 张已填底稿抽出来的，请你重点 review 这部分。包括三级复核、Z6 调整分录编号、7 条事务所内规（AR-RULE-001 应收贷方重分类 / GC-INDICATOR-001 关联担保识别等）。
④ L4 审计师个人经验 (~0%) — 现在还没有，但 Agent 对话面板能捕获你每次的修改。等你用上一段时间，你的判断会自动累积成新的 L3 规则候选。这是事务所专属的"AI 老审计师大脑"。"

📢 跟项目经理讲价值
  • L1+L2 是复用资产 — 我做了一次，下次给立信、信永中和也能用，不重复开发
  • L3 是事务所差异化 — 你的核心 IP，是甲所对外服务的护城河
  • L4 是审计师个人资产 — 跳槽都能带走（如果你愿意），但你愿意把它沉淀到本体里 → 事务所就拥有了"人走茶不凉"
📢 请项目经理 review 的 3 个问题
  1. L2 通用准则版本 — 我用的中注协 2023-12 / 财政部条文等是不是最新？有没有更新过的？
  2. L3 甲所 7 条内规 — AR-RULE-001 / RP-RULE-001 / GC-INDICATOR-001 / DL-RULE-001 等是不是甲所 2024-2025 的实操版本？阈值对不对？narrative 描述准不准？
  3. L3 OT 是否缺漏 — 我从 442-sheet 索引体系抽出了 22 个甲所专有 OT，是不是还有"应该建模但没建"的？（举例：函证流程是否需要单独建 OT？）
