📋 我用的 LLM Prompt 模板（直接复用）

你是审计领域的本体（Ontology）工程师。

请你阅读以下材料：
1. 审计准则：CSA 1101 / 1141 / 1221 / 1311 / 1411（中注协）
2. 这家事务所的 442-sheet 工作底稿模板的 sheet 索引（如 X/Y/A/B/C/D/G/H/Z）
3. 已审计案例：甲公司（通风机械） 2025 年度报表审计

按照 Palantir Foundry 风格 的 ObjectType / LinkType / ActionType 三联建模法，
为这个事务所设计一套审计本体 TBox。要求：

【输出格式】
- ObjectType: 50-100 个对象类型
- LinkType: 70-100 个关系类型
- ActionType: 10-15 个智能体可执行的动作

【设计原则】
1. 按 10 大簇组织 ObjectType：
   客户类 / 项目类 / 数据类 / 审计区域 / 认定 / 风险 / 程序 / 证据 / 底稿 / 报告
2. 每个 OT 必须有：
   - code (PascalCase 英文)
   - display_name (中文)
   - _cluster (上面 10 大簇之一)
   - _layer ("L1" = 国际通用 / "L2" = 中国本地 / "L3:firm" = 本所专有)
   - icon (lucide 图标名)
   - color (16进制色，按簇分类)
   - properties_schema: 字段定义列表 [{code, label, type, required}]
3. LinkType 用数组格式 [code, name, source, target, cardinality]
4. ActionType 必须涵盖 Read/Map/Standardize/Sample/Reconcile/Recompute/Explain/Generate/Review/SignOff/Archive 11 个核心动作
5. 输出符合 W3C OWL 风格的命名约定（OT=类，LT/AT=属性）

请先给出 ObjectType 完整列表（JSON 格式），稍后我会就每个簇内的对象提细化问题。

# build_instances.py 的核心 helper
def add(type_code, key, display_name, data):
    counter[0] += 1
    inst.append({
        "id": counter[0],
        "key": f"{type_code}::{key}",
        "type_code": type_code,
        "display_name": display_name,
        "data": data,
    })
    id_map[f"{type_code}::{key}"] = counter[0]

def link(lt_code, src_key, tgt_key):
    if src_key in id_map and tgt_key in id_map:
        links.append({"link_type_code": lt_code,
                      "source_id": id_map[src_key],
                      "target_id": id_map[tgt_key]})

§8 总结

5 阶段流水线本质：
  1. 阶段 1+2：LLM 引导 + 人工定稿把准则 + 内规变成 TBox + L3 规则库 — 跨客户复用
  2. 阶段 3：openpyxl 脚本 + LLM 抽 PDF 把客户数据变成 L0 ABox — 每客户一次
  3. 阶段 4：Python fill_X 函数显式引用本体填稿 + Provenance.record 记录每格 — 每张底稿一次
  4. 阶段 5：异常 + 复核结构化 — 持续累积
关键洞察：
  • LLM 不参与填稿 — Agent 是确定性 Python，可调试 / 可重跑 / 可审计
  • LLM 只参与 TBox/规则设计 — 即使设计错了也是确定性的（rules 字面量在代码里）
  • 所有数字都有 Provenance — 4250 cell 全部可追溯到 OT/AT/Rule + 原始数据
  • 跨所迁移 = L1+L2 不动 + L3 重抽 + L0 重读 + fill 函数微调
