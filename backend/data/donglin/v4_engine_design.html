v4 本体知识抽取引擎

把方法 A 的业务驱动 TBox 设计 + L1+L2 准则建模 + L0 客户数据 + AuditRule 形式化 与 方法 B 的底稿模板自动抽取 + Turtle/OWL 双轨产物 + evidenceRef 证据链 合并， 再加L4 审计师经验沉淀双管道， 做成统一的、可重跑的、跨所部署的本体知识抽取引擎。
📅 2026-05-24 📂 backend/data/donglin/v4_engine_design.html 🎯 设计阶段（pre-implementation）

📋 TL;DR — 一句话说清 v4 干啥

v4 是一个 6 Phase 自动化管线： Phase 0 人工 + LLM 设计跨所通用 TBox → Phase 1 机器自动扫客户/事务所所有 Excel/VBA/PDF/客户数据出证据 → Phase 2 LLM 给证据打 TBox 标签 → Phase 3 程序生成 ABox + Rule (Turtle + JSON 双轨) → Phase 4 Agent 跑稿带 layer_chain 追溯 → Phase 5 ChatPanel 改写事件 + VBA 模式两路捕获 L4 经验。 每个产物强制带 _layer + _evidence 标签， 跨所部署只换 L3+L0 不动 L1+L2，开发周期从 3 个月 → 1-2 周。

§1 设计目标与非目标

✅ 设计目标 (Goals)
  • 覆盖 4 层：L1 国际 + L2 中国 + L3 事务所 + L0 客户 + L4 经验
  • 双轨产物：Turtle/OWL (审计师 review + SPARQL) + JSON (前端 + 数据库)
  • 证据链强制：每条 L3 ABox / FillRule 必须有 _evidence 指针
  • 跨所可迁移：L1+L2 一字不改，L3 自动重抽，L0 重读客户数据
  • 增量可重跑：检测文件修改时间，只重抽变更部分
  • 可校验：18+ 个 validate.py check 跑过才能 sign-off
  • L4 经验沉淀：ChatPanel + VBA 双管道自动累积候选规则
❌ 设计非目标 (Non-Goals)
  • 不做 PDF OCR — pdftotext 已够；扫描件留给 v5
  • 不全自动化 L3 规则的 formal_expression — 阈值 / 公式仍需审计师人工编码（LLM 只给候选）
  • 不引入 OWL Reasoner — 不跑 HermiT/Pellet 推理，验证用 SHACL-lite
  • 不跨语言 — 只支持中文事务所
  • 不替代审计师判断 — Agent 只做 AI 初稿，最终结论由人审核
  • 不解锁加密 .xlsm — 加密文件 skip + log
  • 不做实时 — 批处理，每个项目跑一次

Phase 2 知识对齐 — LLM 把证据打 TBox 标签

负责人：LLM + 人工 review 边界用例 · 工期：单项目 ~2-4 小时 · 关键创新：替代方法 A 的人工 markdown 步骤

核心算法

# 给每条 jsonl 用 LLM 打 TBox 标签
def align_fact(jsonl_row, tbox_schema):
    prompt = f"""
你是审计本体专家。下面这条证据来自 {jsonl_row['source_file']}，
内容: {jsonl_row}

请判断它属于以下 TBox 类型中的哪一个，并解释原因：
{tbox_schema}

输出 JSON: {{ "tbox_slot": "WorkpaperField" | "Account" | ... ,
            "extracted_fields": {{...}},
            "confidence": 0.0~1.0,
            "_layer": "L1" | "L2" | "L3" | "L0",
            "_layer_basis": "..." }}
"""
    result = llm.complete(prompt, model="claude-opus-4-7")
    return json.loads(result)

# 批量处理 + 缓存
for row in jsonl_rows:
    cache_key = hash(row)
    if cache_key in cache:
        result = cache[cache_key]
    else:
        result = align_fact(row, tbox_schema)
        cache[cache_key] = result
    aligned_facts.append({**row, **result})

# confidence < 0.7 的需人工 review
low_conf = [f for f in aligned_facts if f['confidence'] < 0.7]
human_review(low_conf)  # 写入 review_queue.jsonl
Acceptance Criteria
  • aligned_facts.jsonl 行数 == 输入 jsonl 总行数（不丢数据）
  • 每条带 tbox_slot + _layer + _evidence (原始行号)
  • confidence < 0.7 的 ≤ 5%（高了说明 TBox 设计不全）
  • LLM 调用结果缓存，重跑相同 input 不重复消耗 token

Phase 3 ABox + Rule 生成 — 双轨产物

负责人：程序自动 + 人工编 L2/L3 规则的 formal_expression · 工期：单项目 ~1 天

核心步骤

  1. build_abox.py：把 aligned_facts 按 tbox_slot 分组，每组生成对应类型的 ObjectInstance + LinkInstance
  2. build_fill_rules.py：从 cell_formulas.jsonl + vba_account_map.jsonl 生成 FillRule + WorkpaperField 个体
  3. build_rules_l2.py：手工编 5 条中注协 / 财政部规则的 formal_expression
  4. build_rules_l3.py：手工编 7 条事务所内规（每条来源标 _layer="L3:donglin"）
  5. validate.py：跑 18+ 个一致性 check

validate.py 的 18 个 check（继承 v3）

Check 名称 检查内容

Bilingual labels每个 Class 有 @en / @zh 标签

Acceptance Criteria
  • validate.py 18 个 check 全过
  • 双轨产物 (.ttl + .json) 100% 等价
  • 每条 L3 ABox 都能在 jsonl 中追到证据
  • 规则 formal_expression 可在 sandbox 中 eval 不报错

§9 风险与开放问题

⚠️ R1: LLM 对齐的 confidence 阈值需要校准

Phase 2 的 LLM 打标签可能出现幻觉。需要人工 sample 100 条 review，确认 confidence ≥ 0.7 的准确率 ≥ 95%。如果不达标，需要更细的 prompt 工程 或换更强的模型。

⚠️ R2: 跨所 TBox 复用边界不清

"事务所专有 OT" 和 "中国通用 OT" 的边界主观。如甲所的 AuditArea（按字母分组）— 是 L3 还是 L2？侯佳成可能说这是中注协推荐的通用做法。需要每个 OT 加 _layer_basis 说明依据，由审计师共识决定。

❓ O1: L4 经验沉淀的隐私 / 合规问题

审计师手写的 reason 字段可能含敏感信息（客户身份、内幕判断）。需要决定：是否对 L4 候选规则做匿名化？审计师 sign-off 时是否同意贡献给事务所知识库？这是 Phase 5 实现前必须解决的合规问题。

❓ O2: 双轨产物的同步成本

JSON 和 Turtle 必须严格等价。如果 build_abox.py 有 bug，会导致两者偏离。需要在 validate.py 加 JSON==TTL 这个 check，且每次构建跑 diff。但这增加工程成本，可能不值得。备选：只保留 JSON 单轨，给侯佳成 demo 时单独生成 Turtle 快照。
