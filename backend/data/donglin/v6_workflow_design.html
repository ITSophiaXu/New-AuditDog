v5 分析 + v6 工作流引擎设计

把 v5 的 96 Class 字典 + 我们 review 版的 14 条业务规则 + v4 设计的 4 层标签 / 经验沉淀 合并为统一的 v6 引擎，最终目标是让本体知识库直接驱动甲会计师事务所的年审业务工作流 —— 从计划、执行到报告，一条工作流串起 60+ 张底稿 / 14 条规则 / 4 个 Agent / 三级复核。
📅 2026-05-25 📂 backend/data/donglin/v6_workflow_design.html 🎯 目标：本体驱动业务工作流

📋 TL;DR — 一句话说清

v5 的强项是把 TBox 字典扩展到 96 Class（业务覆盖完整），弱项是没有 AuditRule.formal_expression / `_layer` / `layer_chain` / L4 经验沉淀。
review 版（我们方法 A）的强项是 14 条业务规则带形式化表达式、4,250 cell 追溯能跑通 5 张底稿；弱项是 TBox 只有 52 Class、ABox 人工字面量、不可重跑。
v6 提议：在 v5 的 96 Class + 我们 14 规则的基础上，新增 Layer 6 工作流编排 —— 用 WorkflowDefinition / WorkflowStep / WorkflowExecution 等新 OT，把年审"计划→执行→报告" 的 ~20 步流程结构化，每步绑定底稿 + 规则 + Agent + 复核，让本体直接驱动业务流。

1.3 v5 缺失的关键业务知识层

❌ 1. 没有 AuditRule.formal_expression

v5 的 42 条 FillRule 只有 Excel 公式（"C11 取自 F824"），没有一条业务规则（"应收贷方应该重分类"、"5 笔贷款 100% 关联担保触发持续经营关注"）。

而我们 review 版的 14 条 AuditRule 每条都有 formal_expression（如 closing_cr > 0），Agent 直接可求值触发。

❌ 2. 没有 4 层标签 _layer

v5 的 96 个 Class 全在同一 namespace，没区分 L1/L2/L3。新所部署时不知道哪些可一字不改、哪些必须重抽。

侯老师 review 时也无法快速识别"哪些是国际通用、哪些是甲所自定"。

❌ 3. 没有 cell 级业务追溯链 layer_chain

v5 的 188,955 cell_values 只记录"这格当时是什么数值"，没记录为什么填这个。

我们的 4,250 provenance 每格带 source_kind + ontology_refs + rule_code，能完整追到准则 → 内规 → 客户数据。

❌ 4. 没有自动调整分录建议

v5 有 Z6AdjustmentEntry Class，但没有任何实例（因为没有 AuditRule 触发机制）。

我们 review 版已经自动建议了 2 笔调整：Z6-AI-A6-01 应收贷方 ¥7.11M + Z6-AI-A9-01 出口退税 ¥430K。

❌ 5. 没有 L4 审计师经验沉淀

v5 没有从 ChatPanel / VBA 个性化模式收集审计师经验的管道。

侯总跳槽 / 退休带走的经验**永远沉淀不到事务所知识库**。

❌ 6. 没有业务工作流编排层（最致命）

v5 / Review 版都没有 "年审从计划到报告的工作流" 这一层。底稿 / 规则 / Agent 都各自独立，没有总指挥告诉你"现在该做哪一步、上一步完成后下一步是什么、跨底稿的依赖关系是什么"。

这正是 v6 要补的核心。

分析结论一句话

v5 是一本超详尽的「审计本体词典」（96 Class，但只是描述"世界上有什么"）； Review 版是一本「能跑的审计操作手册」（52 Class + 14 规则 + 5 张能跑的底稿）； 但两者都不是一条「年审业务流水线」 —— 没有"从承接到出报告"的工作流编排。v6 要做的正是把字典和操作手册装到一条流水线里。

§2 v6 设计目标

2.1 终极目标

让甲会计师事务所的审计师能在 v6 引擎上完成一次完整的年审项目： 从承接客户开始，按工作流逐步推进，每一步都由本体知识自动驱动 —— 告诉审计师"现在做什么、用哪张底稿、跑哪条规则、调用哪个 Agent、达到什么条件可签字"。

2.2 三个具体目标

  1. 合并 v5 字典 + Review 版规则 —— v6 TBox = v5 的 96 Class + 我们的 14 条 AuditRule (含 formal_expression)
  2. 引入 v4 设计的 4 层标签 + 经验沉淀 —— 每个 Class 标 _layer，cell trace 带 layer_chain，L4 双管道
  3. 新增 Layer 6 业务工作流编排（v6 核心创新）—— 用 5 个新 OT 描述年审流程：WorkflowDefinition / WorkflowStep / WorkflowExecution / StepResult / Gate

2.3 验证标准

v6 成功标准：
  • 审计师在前端选"甲公司 2025 年审" → 系统自动展开 ~20 步工作流
  • 每步显示"输入底稿 + 触发规则 + 绑定 Agent + 完成条件"
  • Agent 跑稿后自动 mark 该步完成，触发下一步
  • 三级复核签字后该步进入"已完成"状态
  • 整个项目完成时自动生成 AuditReport + Z6 调整分录汇总 + 报告披露段

🎬 场景：甲公司 2025 年审 · A6 应收账款（v6 工作流引擎驱动）
1审计师在前端选 "ENG-JSDW-2025 年审"

v6 的核心价值在这个场景里展现：
  • 审计师不用记得"A6 后面该做什么" — 工作流告诉他
  • Agent 不用人工调度 — 工作流绑定了具体 Agent
  • 规则自动触发 — 工作流知道 A6 该应用哪 3 条规则
  • 复核不会漏 — Gate 强制三级签字才能往下走
  • 经验自动沉淀 — 改写事件累积成下一版规则
